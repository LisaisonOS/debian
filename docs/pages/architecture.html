Architecture System design, overlay structure, and the v1/v2 split

Version History: v1 vs v2

The repository contains two architecture versions. v1 is frozen (no new development); all active work targets v2.

Aspect v1 (Frozen) v2 (Active)
Service management 12 individual systemd user services Single et-supervisor daemon
Mode switching Shell scripts per mode JSON configs in modes.d/
Dashboard Bash + zenity dialogs GTK panel applet with IPC
Configuration Text files + shell scripts JSON profiles + Flask web UIs
Overlays et-v1-general, et-v1-personal et-v2-general, et-v2-personal
Hooks scripts/hooks/v1/ scripts/hooks/v2/

Overlay System

EmComm-Tools uses live-build overlays to inject files into the ISO filesystem. Each overlay maps to /opt/emcomm-tools/ on the installed system.

overlays/ ├── et-v2-general/ # General edition (no personal data) │ └── overlay/ │ ├── etc/skel/ # User home template (copied by useradd -m) │ ├── opt/emcomm-tools/ # Main application tree │ │ ├── bin/ # Scripts, Flask apps, dashboard │ │ ├── conf/ # modes.d/, radios.d/, update.conf │ │ ├── lib/ # et_supervisor Python package │ │ └── sbin/ # Wrapper scripts (rigctld, etc.) │ └── usr/share/ # .desktop files, sudoers, udev rules └── et-v2-personal/ # Personal edition (mirrors general) └── overlay/ # Same structure, may include personal configs
Dual Overlay Rule Changes must go in both et-v2-general and et-v2-personal overlays — they mirror each other. The General edition ships without personal data (callsign, Winlink password); the Personal edition can include pre-configured credentials.

Build Pipeline

The ISO is built with setup-emcomm-iso.sh, a dialog-based interactive script that drives Debian live-build.

setup-emcomm-iso.sh | [Dialog menus] Select: version (v1/v2), edition (general/personal) | [lb config] Set: mirror, architecture, packages, bootloader | [lb build] | +----+----+ | | Hooks Overlays (chroot) (file copy) | | +----+----+ | Live ISO (.hybrid.iso)

Hook Scripts

Hooks run inside the chroot during ISO build. They install software, configure services, and set permissions. Numbered for execution order (e.g., 0100-apt-setup.hook.chroot runs before 0170-install-yaac.hook.chroot).

See Hook Scripts for the full list and best practices.

Runtime Architecture

Process Hierarchy

systemd --user | et-supervisor.service | et-supervisor (Python daemon) | +-- ProcessManager | |-- rigctld (CAT control) | |-- direwolf (TNC) | |-- pat-http (Winlink) | +-- yaac / js8call / wsjtx / ... | +-- ModeEngine | |-- Load JSON mode config | |-- Run prechecks (device, audio, callsign) | |-- Apply config templates | +-- Orchestrate start/stop chain | +-- HealthMonitor | |-- PID liveness checks | +-- TCP port probes | +-- IPCServer |-- Unix socket: /run/user/$UID/et-supervisor.sock +-- JSON request/response protocol

IPC Protocol

The dashboard and CLI communicate with the supervisor via a Unix domain socket using newline-delimited JSON messages.

# Start a mode
{"action": "start", "mode": "winlink-vara-hf"}

# Stop current mode
{"action": "stop"}

# Query status
{"action": "status"}

# Response
{"status": "ok", "mode": "winlink-vara-hf", "state": "running", "processes": [...]}

File Paths at Runtime

Path Description
/opt/emcomm-tools/ Main application tree (read-only on Live Run)
~/.config/emcomm-tools/ User configuration (user.json, custom radios)
~/.config/systemd/user/ Systemd user service files
/run/user/$UID/et-supervisor.sock Supervisor IPC socket
~/Documents/ User documents (persisted to USB on Live Run)
/var/lib/bluetooth/ Bluetooth pairings (requires special handling on Live Run)

Live Run vs HDD Install

EmComm-Tools supports both Live Run (boot from USB, no install) and HDD Install (permanent installation). The key differences affect how data is stored and services are started.

Aspect Live Run (USB) HDD Install
Root filesystem OverlayFS (read-only lower + tmpfs upper) Normal ext4 (fully writable)
Persistence Manual save to USB partition (et-persistence-save) Automatic (normal filesystem)
User services Needs loginctl enable-linger + daemon-reload on login Standard systemd user session
Bluetooth Must rm -rf + recreate /var/lib/bluetooth/ in upper layer Normal operation
WiFi configs Saved/restored via persistence scripts Persistent in /etc/NetworkManager/
Custom radios Saved to ~/.config/emcomm-tools/radios.d/ + symlinked Same approach (works everywhere)
Live Run Overlay Gotcha On Live Run, /opt/emcomm-tools/ exists in the read-only lower layer. Files can appear writable but changes disappear on reboot. Custom files must be written to the user directory and symlinked into the system path. The et-persistence-save script handles saving critical data to USB before shutdown.
