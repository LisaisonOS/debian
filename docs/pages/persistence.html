Persistence Save/restore system for Live Run on USB

The Problem

On Live Run (booting from USB), the root filesystem uses OverlayFS with a tmpfs upper layer. All changes — configuration files, user documents, Bluetooth pairings, WiFi passwords — are lost on reboot. The persistence system solves this by saving critical data to a writable partition on the USB drive before shutdown.

HDD installs don't need this On HDD installs, the filesystem is fully writable and changes persist normally. The persistence scripts detect the install type and skip when not on Live Run (except for et-restore-hw which handles WiFi/BT import on HDD).

USB Partition Layout

USB Drive ├── Partition 1: ISO (read-only) # Live system image └── Partition 2: et-config (writable) # FAT32/exFAT persistence partition └── et-config/ ├── configs/ │ ├── emcomm-tools/ # user.json, callsign, grid, language │ ├── radios/ # Custom radio JSON profiles │ ├── active-radio # Active radio filename │ ├── pat/ # Winlink configuration │ ├── js8call/ # JS8Call settings │ ├── chattervox/ # Chattervox keys │ ├── wifi/ # NetworkManager .nmconnection files │ └── bluetooth/ # bt-pairings.tar (archived) ├── user-data/ │ ├── documents/ # ~/Documents (Winlink PDFs, logs) │ └── downloads/ # ~/Downloads └── update/ # Patch .tar.gz files (offline updates)

Save & Shutdown Menu

Users trigger persistence save via the desktop menu:

Menu EntryScriptAction
Save & Reboot et-save-and-reboot Save all data to USB, then reboot
Save & Shutdown et-save-and-shutdown Save all data to USB, then power off
Save & Suspend et-save-and-suspend Save all data to USB, then suspend to RAM

All three scripts show a progress bar dialog (zenity) with stages:

  1. 10% — Detecting USB persistence partition
  2. 20% — Saving EmComm-Tools config (user.json)
  3. 30% — Saving active radio selection
  4. 35% — Saving custom radio configs
  5. 40% — Saving Pat/Winlink config
  6. 50% — Saving JS8Call settings
  7. 60% — Saving Chattervox keys
  8. 70% — Saving Thunderbird profile
  9. 78% — Saving Documents & Downloads
  10. 85% — Saving WiFi configurations
  11. 90% — Saving Bluetooth pairings
  12. 100% — Complete

Save Script: et-persistence-save

Key behaviors:

  • Detects USB via et-persistence-detect (scans /media/ and /run/media/ for et-config/ directory)
  • Documents/Downloads use rsync --delete --modify-window=2 for FAT-safe incremental sync, with cp -ru fallback
  • Space check with 10% margin before large copies
  • Bluetooth pairings saved as bt-pairings.tar (FAT/exFAT can't store colons in filenames like MAC addresses)
  • sudo tar writes to /tmp first, then user copies to USB (FUSE mounts block root writes)

Restore Script: et-persistence-restore

Runs automatically on boot (autostart desktop entry). Restores data in reverse order:

  1. Custom radio configs (copy + symlink into system dir)
  2. Active radio selection (recreate symlink)
  3. EmComm-Tools config (user.json)
  4. Pat/Winlink, JS8Call, Chattervox settings
  5. Documents & Downloads (rsync without --delete to merge, not overwrite)
WiFi & Bluetooth on HDD et-persistence-restore skips WiFi/BT on HDD installs (they persist natively). To import WiFi/BT configs from USB to a new HDD install, use sudo et-restore-hw (standalone script with sudoers entry).

Known Gotchas

exFAT/FAT32 Limitations
  • No colons in filenames — BT MAC addresses must be archived, not stored as directory names
  • cp -a fails — FAT doesn't support Unix ownership/xattrs, so -a flag returns errors
  • 2-second timestamp resolution — use rsync --modify-window=2 for accurate sync
OverlayFS Lower Layer /var/lib/bluetooth/ exists in the ISO's read-only lower layer. Writes appear to succeed (tar shows extracted files) but nothing persists. Must sudo rm -rf /var/lib/bluetooth && sudo mkdir -p /var/lib/bluetooth to recreate in the writable upper layer before restoring pairings.
FUSE Mount Permissions sudo cannot write directly to FUSE-mounted USB partitions. And sudo commands fail with "getcwd: cannot access parent directories" when the working directory is on a FUSE mount. Solution: write to /tmp with sudo, then copy to USB as the regular user.
